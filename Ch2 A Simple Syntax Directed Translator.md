## 2.2 Syntax Definition

```
if (expression) statement else statement
```

```
stmt -> if (expr) stmt else stmt
```

such a rule is called a production

production: lexical elements like if and () are called terminals, variables like expr and statement are called non-terminals

## 2.2.1 Definition of Grammars

context-free grammar has 4 components:
1. set of terminal symbols (tokens)
	1. additional info in token on attribute values is not part of grammar, so we call tokens terminals
2. set of non-terminals (syntactic variables), set of strings of terminals
3. set of productions, left side/head is a nonterminal + right side/body of arrow is a sequence of terminals/non-terminals
	1. can have multiple bodies for a production using |
4. one non-terminal is designated as the start symbol

## 2.2.2 Derivations

parsing: taking a string of terminals + figuring out how to derive it from the start symbol of the grammar
- if it can't be derived from the start symbol of the grammar then report syntax errors withing string

## 2.2.3 Parse Trees

parse trees shows how start symbol of a grammar derives a string in the language

given a CFG, a parse tree according to the grammar is a tree with the following properties

1. root is labeled by the start symbol
2. each leaf is labeled by a terminal or e
3. each interior node is labeled by a nonterminal
4. if A is the nonterminal labeling some interior node and X1, X2, ..., Xn are the labels of the children of that node from left to right then there must be a production A -> X1, X2, ..., Xn

- from left to right, leaves of parse tree form yield of tree— string generated or derived from the nonterminal at the root of parse tree

# 2.3 Syntax-Directed Translation

- attributes: any quantity associated with a programming construct
	- ex: data types of expressions, number of instructions in generated code, location of first instruction in generated code
- (syntax-directed) translation schemes: notation for attaching program fragments to the productions of a grammar
	- program fragments are executed when production is used during syntax analysis
	- combined result of fragment executions in order induced by syntax analysis produces translation of program to which analysis/synthesis process is applied

## 2.3.1 Postfix Notation

postfix notation for expression E can be defined inductively:
1. if E is a variable or constant, then the postfix notation for E is E itself
2. if E is an expression of the form $E_1$ op $E_2$, then the postfix notation for E is $E_1'$ $E_2'$ op where $E_1'$ and $E_2'$ are the postfix notations for $E_1$ and $E_2$ 
3. if E is of the form $(E_1)$ then the postfix notation for $E$ is the same as the postfix notation for $E_1$

## 2.3.2 Synthesized Attributes

syntax-directed definition associates:
1. with each grammar symbol, a set of attributes
2. with each production, a set of semantic rules for computing the values of the attributes associated with the symbols appearing in the production

attribute is synthesized if its value at a parse-tree node N is determined from attribute values at children of N + at N itself
- can be evaluated during a single bottom-up traversal of a parse tree

## 2.3.4 Tree Traversals

traversal: starts at root + visits each node of the tree in some order

depth-first traversal: starts at root + recursively visits each node in any order

synthesized attributes can be evaluated during any bottom-up traversal
- traversal that evaluates attributes at a node after having evaluated attributes at its children

## 2.3.5 Translation Schemes

preorder traversal: action is done when we first visit a node
postorder traversal: action is done just before we leave a node for the last time

implementation of translation scheme must ensure that semantic actions are performed in the order they would appear during a postorder traversal of a parse tree

# 2.6 Lexical Analysis

lexical analyzer: reads characters from the input + groups them into token objects

a token is a terminal along with additional info

lexeme: sequence of input characters that comprises a token

lexical analyzer insulates a parser from the lexeme representation of tokens

## 2.6.1 Removal of White Space and Comments

if white space is eliminated by the lexical analyzer, the parser will never have to consider it

## 2.6.2 Reading Ahead

approach to reading ahead on input: maintain input buffer from which lexical analyzer can read and push back characters

pointer keeps track of portion of input that's been analyzed

peak: 1-character read-ahead

lexical analyzer reads ahead only when it must (ex: not for *)

invariant assertion: when lexical analyzer returns a token, peek holds character beyond lexeme for current token, or it holds a blank

## 2.6.3 Constants

int constants can be allowed by either creating a terminal symbol (**num**) for such constants or by incorporating the syntax of integer constants into the grammar

job of collecting characters into integers + computing their collective numerical value is usually given to a lexical analyzer, so numbers can be treated as a single unit during parsing + translation

## 2.6.4 Recognizing Keywords and Identifiers

char string forms identifier only if it's not a keyword

grammars treat identifiers as terminals to simplify parser, so it can expect same terminal (ex: id), each time any identifiers appears in the input

lexical analyzer solves these problems by using a table to hold character strings:
- single representation: phases of compiler can work with refs or pointers to string in table
- reserved words: string table initialized with reserved strings + tokens

## 2.8.3 Static Checking

static checks = consistency checks done during compilation

- syntactic checking: there's more to syntax than grammars like other syntactic constraints:
	- identifier being declared at most 1x in a scope
	- break statement must have enclosing loop or switch statement
- type checking: assure that an operator or function is applied to the right number + type of operands

### L-values and R-values

simple static checks that can be done during construction of a syntax tree for a source program

l-values + r-values: values that are appropriate on the left and right sides of an assignment
- l-values = locations
- r-values = values
- i = 5
- i = i + 1

static checking must assure that the left side of an assignment = l-value (i, a[2], ..., not r-value like 2)

### Type Checking

in `if (expr) stmt`, `expr` is expected to have type boolean
idea of matching actual with expected type applies for:
- coercions: type of operand automatically converted to type operator expects
- overloading: for ex, + operator means addition for ints + concatenation for strings
